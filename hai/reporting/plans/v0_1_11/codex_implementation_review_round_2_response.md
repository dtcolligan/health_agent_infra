### F-IR2-01. W-E fingerprint is timestamp-sensitive, so same-state reprojection can auto-supersede

**Q-bucket:** Q1 / Q3
**Severity:** blocker
**Reference:** `src/health_agent_infra/core/synthesis.py:417`, `src/health_agent_infra/core/synthesis.py:431`, `src/health_agent_infra/core/synthesis.py:456`, `src/health_agent_infra/core/state/projector.py:440`, `src/health_agent_infra/core/state/projector.py:503`, `src/health_agent_infra/core/state/projector.py:863`, `src/health_agent_infra/core/state/projector.py:897`, `verification/tests/test_daily_supersede_on_state_change.py:323`
**Argument:** The round-1 fix closes the specific “proposal rows unchanged while accepted state changed” hole, but it does so by hashing `projected_at` and `corrected_at` from each `accepted_*_state_daily` table. That conflicts with the W-E/Q3 contract that wall-clock churn must not flip the state fingerprint. The function documents both sides of the contradiction: it says accepted-state timestamps are hashed because “These update on every UPSERT” (`synthesis.py:417-424`), then says wall-clock fields are excluded because they churn without substantive content changes (`synthesis.py:431-432`). The normal daily path can reproject the same raw/clean state before synthesis (`cli.py:4795-4804` via `_daily_pull_and_project`), and the projectors update those timestamps on ordinary existing-row updates (`projector.py:440-483`, `projector.py:503-514`, `projector.py:863-878`, `projector.py:897-908`). That means a same-content reprojection can produce a different fingerprint and mint `_v<N>` even though the semantic synthesis inputs did not change. The new acceptance test only proves that changing nutrition content plus `corrected_at` triggers `_v2`; it does not prove that identical content with refreshed projection timestamps remains a true no-op.
**Recommended response:** Rework `_compute_state_fingerprint` to hash semantic accepted-state content, not projection/correction wall-clock columns. The hash should include stable content-bearing fields such as domain values, derivation path, source, ingest actor, and stable `derived_from`/source identifiers as appropriate, while excluding `projected_at`, `corrected_at`, `produced_at`, and other reload timestamps. Add a regression test that writes the same accepted-state content with different projection/correction timestamps and asserts the second synthesis returns the existing canonical plan id with no new `daily_plan` row. Keep the state-change test, but make the changed value itself the reason the fingerprint flips.

### F-IR2-02. Demo regression proof still overstates the runnable v0.1.11 flow

**Q-bucket:** Q4 / Q6
**Severity:** important
**Reference:** `reporting/plans/v0_1_11/RELEASE_PROOF.md:129`, `reporting/plans/v0_1_11/RELEASE_PROOF.md:172`, `reporting/docs/demo_flow.md:144`, `reporting/docs/demo_flow.md:172`, `reporting/docs/demo_flow.md:183`, `reporting/plans/v0_1_11/PLAN.md:1217`, `verification/tests/test_demo_isolation_surfaces.py:168`
**Argument:** The round-1 response honestly changes the canonical CLI-only demo path to stop at `overall_status: "awaiting_proposals"` and narrate the runtime/skill boundary (`demo_flow.md:144-160`, `demo_flow.md:172-187`). That is coherent as a product/demo decision, but the ship-gate and proof were not fully propagated. `RELEASE_PROOF.md` still says the manual seed populates enough state for `hai today` to render and that `hai daily --skip-pull --source csv` synthesizes plus re-run auto-supersedes `_v2` (`RELEASE_PROOF.md:129-135`), while the actual transcript stops at `awaiting_proposals` (`RELEASE_PROOF.md:172-176`). PLAN.md still says isolation-replay mode populates enough state for `hai today` to render and replays § B end-to-end (`PLAN.md:1217-1224`). The new subprocess isolation test is useful, but it exercises only `demo start`, three `intake` commands, and `demo end`; it does not run `hai daily`, `hai today`, `hai intake gaps --from-state-snapshot`, `hai doctor --deep`, or the fresh-date `hai daily --supersede` refusal while checksumming real state (`test_demo_isolation_surfaces.py:227-270`). The proof therefore claims a broader runnable gate than the transcript/test actually establish.
**Recommended response:** Either restore a real path (likely manual proposal seeding) that satisfies the existing PLAN/RELEASE_PROOF bullets, or explicitly revise the ship gate and proof to define v0.1.11 isolation replay as a boundary-stop demo, with `hai today` showing no plan by design. Whichever contract is chosen, update the transcript and permanent subprocess test so they exercise the named commands that remain in the gate.

### F-IR2-03. W-W scope narrowing is only partially reflected in PLAN.md and tests

**Q-bucket:** Q2 / Q3 / Q6
**Severity:** important
**Reference:** `reporting/plans/v0_1_11/PLAN.md:863`, `reporting/plans/v0_1_11/PLAN.md:876`, `reporting/plans/v0_1_11/PLAN.md:915`, `reporting/plans/v0_1_11/PLAN.md:919`, `reporting/plans/v0_1_11/PLAN.md:927`, `reporting/plans/v0_1_11/PLAN.md:950`, `verification/tests/test_intake_gaps_from_snapshot.py:159`, `verification/tests/test_intake_gaps_from_snapshot.py:192`, `src/health_agent_infra/core/intake/gaps.py:287`
**Argument:** The round-1 response narrows W-W away from JSONL-tail consistency and toward SQLite-only snapshot derivation (`PLAN.md:876-884`), which is a reasonable acceptance if fully propagated. It is not fully propagated. The same W-W section still lists new `test_intake_gaps_source_parity.py`, `test_intake_gaps_concurrency.py`, `test_intake_gaps_jsonl_old_rows_kept.py`, and `test_intake_gaps_capabilities_emission.py` files (`PLAN.md:915-936`) even though the branch only has `test_intake_gaps.py` and `test_intake_gaps_from_snapshot.py` for this surface. The acceptance line still says the concurrency test is green and “100/100 trials produce one of two valid shapes” (`PLAN.md:950-951`), but the landed test is sequential determinism over an unchanged DB, not a concurrent write/read assertion (`test_intake_gaps_from_snapshot.py:159-189`). Also, PLAN.md says snapshot derivation refuses if no successful `sync_run_log` entry exists within 48h (`PLAN.md:863-865`), while implementation/test explicitly allow no sync history to pass (`gaps.py:287-318`, `test_intake_gaps_from_snapshot.py:192-203`). Those are contract/documentation mismatches, not just missing polish.
**Recommended response:** Finish the narrowing in one direction. If SQLite-only/no-history-ok/sequential-determinism is the intended v0.1.11 contract, revise PLAN.md, RELEASE_PROOF, and test names/comments to say that explicitly and remove the stale promised files/100-trial concurrency claim. If the original contract stands, add the missing parity/capabilities tests and a real concurrent write/read test, and make no sync history refuse unless `--allow-stale-snapshot` is passed.
