## 1. Why this doc exists

The project ships test code in three places — `verification/tests/`
(unit + integration), `src/health_agent_infra/evals/scenarios/`
(per-domain + synthesis fixtures with rubrics), and (as of v0.1.10)
`verification/dogfood/` (persona-driven regression). Each has a
purpose; no doc has yet explained how they fit together.

This matters because:
- **Test coverage and eval coverage are different.** A 100% line-
  covered codebase can still produce wrong recommendations for
  whole user-shape segments. v0.1.10's persona harness proved this
  empirically (F-C-03 hidden behind passing unit tests).
- **The runtime makes claims an LLM-judge can falsify.** Weekly
  review prose, rationale text, and uncertainty surfacing are
  load-bearing on user trust. Validating them against the
  underlying state DB requires a separate evaluation loop.
- **The deferred W58 factuality gate is non-trivial to scaffold.**
  It needs a model, a rubric, an agent-judge negotiation loop, and
  per-week test coverage. Doing this right requires upstream eval
  decisions captured here.

## 2. The five eval classes

| Class | What it tests | Where it lives | Run frequency |
|---|---|---|---|
| **Unit** | Individual functions / classes produce correct outputs | `verification/tests/` | Every commit (CI) |
| **Scenario** | Per-domain classify + policy paths produce expected bands + decisions on hand-crafted inputs | `src/health_agent_infra/evals/scenarios/<domain>/*.json` + `evals/runner.py` | On-demand via `hai eval run` |
| **Persona** | Full pipeline produces sane output for synthetic user archetypes spanning age/sex/sport/source/history | `verification/dogfood/` | Manual on release; future: subset in CI |
| **Outcome / calibration** | Confidence values track ground-truth helpfulness over time | NOT YET BUILT — v0.2+ | Weekly post-v0.2.0 |
| **Adversarial / red-team** | System fails gracefully on malformed inputs, edge cases, attack-shaped queries | PARTIAL — some in unit + scenario; no dedicated suite | On-demand |

Each class has a different observation surface:
- Unit observes Python return values.
- Scenario observes classified_state + policy_result.
- Persona observes the rendered `today` text + `explain` JSON.
- Outcome observes user-reported review_outcome rows over time.
- Adversarial observes whether the system crashed, validated, or
  produced wrong-but-plausible output.

A bug class can hide in the gap between two eval classes. F-C-03
hid in the gap between unit (passed — projection function works in
isolation) and persona (revealed — projection wasn't being invoked
in the clean flow).

- 2169 tests passing as of v0.1.10 ship.
- Coverage by package surface: well-covered (every projector,
  classifier, policy entry, intake handler has dedicated tests).
- Coverage by data shape: weaker. Most tests assume happy-path
  inputs; few exercise None-propagation, threshold-boundary,
  malformed-fixture cases.

### 3.2 Scenario

|---|---|---|---|
| recovery | 3 | classified_bands, policy_decisions | skipped (no skill harness) |
| running | 3 | classified_bands, policy_decisions | skipped |
| sleep | 3 | classified_bands, policy_decisions | skipped |
| stress | 3 | classified_bands, policy_decisions | skipped |
| strength | 3 | classified_bands, policy_decisions | skipped |

Per-domain scenario count is thin. Three scenarios cover roughly:

### 3.4 Outcome / calibration

**Not yet built.** Requires v0.2.0 weekly review + insight ledger
to accumulate the (recommendation, compliance, outcome) triples
that make calibration measurable.

### 3.5 Adversarial / red-team

**Partial.** Unit tests exercise some malformed inputs (e.g.
validator rejection tests). No dedicated red-team suite.

## 4. Coverage gaps

Largest gaps per eval class, ordered by strategic impact:

### 4.1 Persona × scenario interaction (largest gap)

The scenario fixtures are defined against generic inputs; the
persona harness drives synthetic pipeline runs. There is no
explicit assertion that *for persona P2 (32yo female marathoner),
the running domain produces action X under condition Y*. Persona
expected-behaviour is currently auto-detected via heuristics in
`runner._synthesise_findings`, not declared.

**Fix scope.** v0.1.14 W-AK — per-persona expected-behaviour
assertions. Each persona's spec gains an `expected_actions`
dict mapping `{date_offset: {domain: action}}`. The harness
asserts equality.

### 4.2 Threshold-boundary tests

Most scenarios exercise the "obvious" cases (clearly rested vs.
clearly fatigued). Few exercise the boundary (e.g., HRV exactly at
the band threshold). Boundary bugs hide here.

**Fix scope.** v0.1.11 W-P (property-based tests via Hypothesis)
addresses this generically. v0.1.14 W-AH (scenario expansion) adds
explicit boundary scenarios.

### 4.3 Multi-rule interaction

Most scenarios fire one rule. The synthesis surface fires multiple
X-rules sometimes. Coverage of multi-rule interaction is light.

**Fix scope.** v0.1.14 W-AH expansion includes synthesis scenarios
where 2+ X-rules interact.

### 4.4 Skill-layer prose

Every scenario rubric marks Axis 3 (rationale_quality) as
`skipped_requires_agent_harness`. The skill prose is never
evaluated. v0.2.0 W58 will close this for the weekly review;
the daily-recommendation skill prose stays unevaluated until at
least v0.3+.

### 4.5 Cross-day state transitions

Scenarios are single-day snapshots. Persona harness drives
multi-day pipelines but evaluates only as-of-today output. Tests
asserting "after a 14-day gap, the recovery domain should
re-classify within N days of resumed signal" do not exist.

**Fix scope.** v0.1.11 W-O P12 (vacation-returner persona) plus
v0.1.14 W-AK (per-persona expected behaviour) closes this for the
acute case.

### 4.6 Vendor variability

Real Garmin / intervals.icu pulls vary in field availability per
account, device generation, and sync cadence. Synthetic personas
use deterministic inputs. Tests against actual upstream variability
do not exist.

**Fix scope.** Long-tail; defer until vendor adapter churn forces
the issue. Captured in `risks_and_open_questions.md`.

## 5. Ground-truth methodology

A test is only as good as its expected output. For a
recommendation system, "expected" is non-trivial.

### 5.1 Layered ground truth

| Layer | Source of truth | Confidence |
|---|---|---|
| **Mechanical** | The runtime *should* produce X according to the documented R-rule / X-rule. | High (it's a derivation question). |
| **Sport-science consensus** | The runtime *should* downgrade hard sessions when ACWR > 1.5. | Medium (consensus but not fully specified). |
| **N-of-1 outcome** | The recommendation *should have helped* this specific user. | Low (single-user, biased reporting). |
| **Population outcome** | The recommendation *should help on average* across users. | Not yet measurable (no community). |

The four classes have different evaluation surfaces:

- Mechanical → unit tests, scenario rubrics. Black-and-white.
- Sport-science consensus → scenario rubrics with citation. Reviewed
  by maintainer at scenario-authoring time.
- N-of-1 outcome → review_outcome rows. v0.2.0 substrate.
- Population outcome → community telemetry. Post-v1.0 if ever.

### 5.2 Maintainer-as-oracle for v0.1.x

For v0.1.x, the maintainer is the ground-truth oracle for
mechanical + sport-science classes. The eval review tool
(v0.1.14 W-AI) lets the maintainer flag a scenario's expected
output as "correct per spec," "correct per consensus," or
"disagree, here's why." Disagreements either reshape the rubric
OR document the runtime as deliberately conservative.

### 5.3 Future N-of-1 oracle

Post-v0.5 substrate ship, the (recommendation, compliance,
outcome) triple ledger gives a *user-grounded* oracle. The
calibration eval (post-v0.2.0) compares confidence levels to
self-reported outcome correlation.

## 6. LLM-as-judge approach

W58 (deferred from v0.1.9, now v0.2.0 wave per strategic plan)
introduces an LLM-judge factuality gate for the weekly review.

### 6.1 Decision boundary

The judge evaluates **only one rubric**: every quantitative claim
in the weekly review is derivable from cited state-DB rows. Out
of scope: tone, helpfulness, clinical accuracy.

### 6.2 Model choice

Local Prometheus-2-7B (or a comparable open-weight model). Pinned
by SHA in `pyproject.toml` or a separate `judge_model.lock` file.
No upgrade without a release cycle.

### 6.3 Negotiation loop

Per Decision 9 (settled 2026-04-25):
1. Judge flags specific unsupported claims.
2. Flag goes back to the *agent*, not the user.
3. Agent must either correct the claim and resubmit OR defend the
   original claim with explicit evidence the judge missed.
4. The negotiation exchange (claim, flag, response) is logged
   alongside the final summary.
5. Maximum 2 round-trips; if convergence fails, the review is
   delivered with an explicit "judge-disputed" flag.

### 6.4 What this is NOT

- NOT a clinical-claim filter. Banned-token sweep handles that.
- NOT a helpfulness rubric. Out of scope.
- NOT applied to daily recommendations in v0.2.0. Only weekly
  review. Daily-rec evaluation deferred.

### 6.5 Hardware budget

Apple silicon M-series at the maintainer's typical generation runs
Prometheus-2-7B at ~25-40 tok/s. Weekly review prose is ~500
tokens; one judge pass + one negotiation round = ~2-4 seconds.
Acceptable for a once-a-week trigger.

## 7. Persona-driven eval

The dogfood harness is the most strategic eval surface today
because it catches what other classes cannot.

### 7.1 What persona eval catches uniquely

- **Classifier-assumption bugs** — when a band threshold makes
  sense for one user shape but not another (e.g., resting HR 48 is
  athletic for P1, but resting HR 48 for an older recreational
  user might indicate over-training).
- **Cold-start path bugs** — P8 (day-1 fresh install) probes the
  zero-history path, which is hard to keep covered in unit tests.
- **Cross-domain interaction in realistic shapes** — synthesis
  scenarios cover X-rules in isolation; persona runs catch
  interactions that emerge from a specific user's combined state.
- **Out-of-band assumptions** — P10 (adolescent) deliberately
  probes the user-set boundary.

### 7.2 What persona eval does NOT catch

- Single-function correctness (unit covers).
- Specific R-rule firings on hand-crafted inputs (scenario covers).
- Outcome calibration over time (needs v0.2+ substrate).
- Skill prose quality (W58 covers eventually).

### 7.3 Expansion roadmap

| Release | Persona count | What's added |
|---|---|---|
| v0.1.10 (shipped) | 8 | P1-P8 baseline matrix |
| v0.1.11 | 12 | + P9-P12 (older female endurance, adolescent boundary, elevated stress, vacation returner) |
| v0.1.14 | 12 + assertions | Per-persona `expected_actions` declared |
| v0.2.0 | 14-16 | + multi-day outcome trajectories per persona |
| v0.4 | 16-20 | + personas representing second user (post-MCP) |

Personas are not free — each represents an authoring + maintenance
cost. Expansion stops when the marginal persona doesn't surface
new failure modes.

## 8. Adversarial eval

A dedicated red-team suite is **not yet scoped** (post-v0.1.13
per tactical plan). Some adversarial coverage exists in unit
tests:

- Validator rejection tests (banned-token sweep, schema-version
  mismatch, action-enum mismatch).
- Confirm-required tests (W57 runtime gate).
- USER_INPUT exit-code tests (hai capabilities --json missing,
  malformed JSON).

What's missing:

- **Malformed-evidence resilience** — does `hai clean` handle
  truncated CSV, NaN values, future-dated rows?
- **Threshold-injection** — does the runtime tolerate user
  thresholds.toml with malicious shapes (deeply nested keys,
  recursive includes, oversized lists)?
- **State-DB corruption** — does `hai doctor` recover or fail
  cleanly when state.db has partial-write corruption?
- **Concurrent mutation** — does running two `hai intake` commands
  simultaneously produce a clean state or a deadlock?

These belong in a v0.1.13+ red-team workstream
(`tactical_plan_v0_1_x.md` § 4 mentions error-message quality
pass; this is the related-but-deeper sibling).

## 9. Calibration eval

Calibration eval is the fundamental measure of trust: when the
agent says `confidence: high`, are users *more often* satisfied
with the recommendation than when it says `confidence: low`?

### 9.1 Why this matters

A health agent that confidently recommends wrong things is
worse than one that abstains. Calibration is the bridge from
"the runtime computes a confidence" to "the user can trust that
confidence to mean something."

### 9.2 Why it's not yet measurable

Requires:
- Recommendation rows with confidence values (have).
- Outcome rows with self-reported helpfulness (have, but sparse).
- ≥ 90 days of paired data (do not have, will have post-v0.5).

### 9.3 What it'll look like

For each domain, compute the correlation between
`recommendation.confidence` and
`review_outcome.self_reported_improvement` (boolean) over a
trailing 90-day window:

- **Well-calibrated:** confidence-high recommendations correlate
  > 0.5 with self-reported improvement; confidence-low correlate
  ≤ 0.2.
- **Overconfident:** confidence-high correlates ≤ 0.5.
- **Underconfident:** confidence-low correlates > 0.5.

Both miscalibration directions are bugs; the runtime should
adjust the rule firings producing them.

### 9.4 First measurement

Post-v0.5 ship + 90 days of substrate. Earliest: 2027-Q2 in
optimistic case.

## 10. Eval surface CLI

### 10.1 Today

- `hai eval run --scenario <path>` — run a single scenario file.
- `python -m verification.dogfood.runner` — run persona harness.

### 10.2 v0.1.14 additions (W-AN)

- `hai eval run --scenario-set <set>` — batch eval.
- `hai eval list` — list scenario sets + counts.
- `hai eval calibration --days 90` — report confidence-vs-outcome
  correlation per domain (no-op if substrate < 90 days; fails
  loudly).

### 10.3 v0.2.0 additions

- `hai review weekly` — gates on factuality judge.
- Judge SHA + score logged in review row.

## 12. Open questions

These are eval-strategy decisions the maintainer hasn't yet
made. Each carries a "decision needed by" milestone.

### Q1. Should the persona harness run in CI?

**Status quo:** Manual invocation on release. Full matrix takes
minutes; CI stays fast.

**Question:** Should a 2-3 persona subset run on every commit?

**Trade-off:** CI speed vs. catching persona regressions early.

**Decision needed by:** v0.1.12 review (W-Y in tactical plan).

### Q2. What's the right scenario count per domain?

**Status quo:** 3 per domain. v0.1.14 W-AH targets 30+ total
new scenarios, distributed roughly 4-6 per domain.

**Question:** Is there a diminishing-returns point? When does a
new scenario stop adding signal?

**Heuristic:** stop adding when the last 5 scenarios all confirm
already-tested behaviour (no new band/rule combinations exercised).

### Q3. Should the LLM judge be cloud or local?

**Status quo (per D1, settled 2026-04-25):** Local Prometheus-2-7B.

**Question:** Should the v0.2.0 ship support cloud Anthropic API
as a fallback for users without the hardware budget?

**Trade-off:** local-first invariant (H2) vs. accessibility for
maintenance-resource-constrained users.

**Default:** local-only at v0.2.0. Revisit at v0.3 if user feedback
indicates the hardware bar is too high.

### Q4. Should adversarial eval be a separate suite or distributed in unit/scenario?

**Status quo:** Distributed; some adversarial cases live in unit,
some in scenario, no dedicated red-team folder.

**Question:** Concentrate into `verification/adversarial/`?

**Trade-off:** Discoverability (concentrated wins) vs. proximity
to the code being attacked (distributed wins).

**Decision needed by:** v0.1.13 W-AD (error-message quality pass)
or v0.1.14 W-AM (adversarial scenarios).

### Q5. How do we evaluate skill prose without invoking skills?

The scenario rubric Axis 3 is `skipped_requires_agent_harness`
because skills are markdown protocols Claude reads. Running them
in CI means running an LLM in CI.

**Options:**
- a. Skip entirely until v0.2.0 W58 (current state).
- b. Mock the skill output for testing — but this defeats the
     purpose.
- c. Snapshot skill prose for known scenarios; compare against
     committed baselines.
- d. Run a local LLM in CI for skill prose only (slow but
     deterministic).

**Default:** (a) until v0.2.0; reconsider at v0.2.0 review.
