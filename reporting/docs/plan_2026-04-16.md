## A. Project interpretation

Health Agent Infra (internal name: Health Lab) is a **governed runtime and contract layer** that turns user-owned health evidence into structured state, making safe, personally tailored agent action possible. The controlling thesis is [canonical_doctrine.md §Thesis](canonical_doctrine.md#thesis):

> Health Lab is the governed runtime that turns user-owned health evidence into structured state, making safe, personally tailored agent action possible.

The wedge is **not** a chatbot, a wearable API, or a broad AI health app. It is the narrow, currently under-built layer between raw personal evidence and bounded agent action — specifically: **evidence → state → policy → recommendation → action → review**. The project's value is both governance and **state-conditioned personal tailoring**; most systems try to personalize at the chat layer, Health Lab personalizes at the state-to-action layer. See [canonical_doctrine.md §Doctrine on state](canonical_doctrine.md#doctrine-on-state) and §Doctrine on policy.

This phase is about producing a single credible public proof of that thesis, not expanding surface area. See the five operating directives in [canonical_doctrine.md §Five operating directives](canonical_doctrine.md).

## B. Definition of success

The current phase succeeds if Health Lab exposes **one narrow, inspectable, end-to-end loop that turns user-owned health evidence into structured state and then into a safe, personally tailored, reviewable action.**

Concretely, the flagship loop [`recovery_readiness_v1`](flagship_loop_spec.md) must satisfy all seven non-negotiable criteria from the founder brief:

1. One real end-to-end loop (`PULL → CLEAN → STATE → POLICY → RECOMMEND → ACTION → REVIEW`) — proven in `clean/health_model/recovery_readiness_v1/` with the CLI at `cli.py`.
2. Explicit state object with uncertainty and missingness fields — see [state_object_schema.md](state_object_schema.md).
3. Executable policy (not prose) — see [minimal_policy_rules.md](minimal_policy_rules.md) and `clean/health_model/recovery_readiness_v1/policy.py`.
4. Structured recommendation with confidence / uncertainty / follow-up — see [recommendation_object_schema.md](recommendation_object_schema.md).
5. One bounded action path — writeback to local JSONL + daily plan markdown, locality enforced.
6. Review mechanism — scheduled review events, outcomes linked to recommendation.
7. Legibility — README, flagship walkthrough, and inspectable proof artifacts.

The current proof bundle at `reporting/artifacts/flagship_loop_proof/2026-04-16-recovery-readiness-v1/` meets 1–6 and partially meets 7 (the legibility tightening is W2 in this plan).

## E. Dependency-aware roadmap

```
W1 (this doc, skeleton) ────┐
                            ├──► W3a · W3b · W3c ──► W4
W2 (README polish)      ────┘
                            │
W1 finalisation ────────────┘  (after W3 scope locks)
```

- **W1 + W2 in parallel**: no code dependencies; different files; both cheap.
- **W3 blocks W4**: the calibration stub (W3b) is structural logic that must be verified on the clean synthetic signal before real noisy Garmin data is introduced. Doing W4 first couples correctness of the stub to data-quality debugging.
- **W3a internally blocks W3c**: new fixture pair needs existing; tests can follow.
- **W4 is adapter-only**: it is gated but not complex — it should not change any schema, policy, recommendation, action, or review surface. If it tries to, that is a scope-discipline violation.

Parallelisable work: W1 and W2 can run simultaneously; within W3, a/b/c can be staged within one sitting.

## F. Non-goals

Controlling doc: [explicit_non_goals.md](explicit_non_goals.md). The full list is 12 hard non-goals and 2 soft non-goals. Reaffirmed for this plan:

- **No second connector.** Garmin plus typed manual readiness intake remain the only inputs.
- **No rich UI / web dashboard / mobile app.** CLI + file artifacts are the proof surface.
- **No medical-style outputs.** No diagnosis, no condition naming, no clinical authority.
- **No deep nutrition system.** Nutrition is not an input of this loop.
- **No new MCP servers or transport layers.**
- **No cosmetic repo refactor.** The eight-bucket layout is preserved; the 7-part runtime is a conceptual overlay, not a reorganisation target.
- **No fictional personas in tailoring scenarios.** Tailoring is demonstrated as goal-conditioned variance on identical evidence, not by inventing users.
- **No learning loop / ML model for W3b calibration.** Stub function only.
- **No live Garmin pull as a committed dependency.** Real data enters the repo as a redacted captured artifact, not as a scheduled or live integration.

If any W-item appears to pull scope toward one of these, it is a distraction and should be deferred.

## G. Risks

Mapped from the founder brief §13 failure modes to project-specific mitigations:

| Failure mode (brief §13) | Project-specific mitigation |
|---|---|
| **1. Abstraction inflation** — sophisticated talk, thin implementation | W1 is an index with links, not prose re-derivation. W3b is a stub with verifiable logic, not a framework. |
| **2. Connector sprawl** | W4 is an adapter-only slice; one source, one day, one captured artifact. No scheduled pulls. |
| **3. Implicit state** | `RecoveryState` remains the single typed object feeding RECOMMEND; W3 does not introduce prose-based state. |
| **4. Policy only in prose** | All 6 rules in `policy.py` remain executable and test-covered; W3 adds no prose policy. |
| **5. No review** | REVIEW is already implemented and exercised by all 6 scenarios; W3b extends it structurally, not cosmetically. |
| **6. Repo legibility failure** | W2 directly addresses: sharp thesis opener, 3-minute read path, CONTRIBUTING surfaced. |
| **7. UI distraction** | W2 adds text-only content. No image assets, no diagrams beyond optional ASCII. No frontend work. |

Additional plan-specific risks:

- **W3a drifts into persona invention.** Mitigation: tailoring scenario pair uses identical evidence with different `active_goal` only; one founder, two goals.
- **W4 credential leakage.** Mitigation: captured JSON is redacted; credentials stay in env vars; no `.env` committed.
- **W3a breaks the existing 6 scenarios.** Mitigation: additive only — add the 7th scenario, regenerate its artifacts, leave the other 6 untouched.
